## 0. 한 줄 요약

킨더보드가 "문서 단위"였다면 **킨더버스는 "공간 단위" 워크스페이스**다. 교사가 자연어로 의도를 말하거나 보드 위 대상을 선택해 명령하면, 얇은 3계층 에이전트가 의도를 해석해 전문 에이전트·도구로 라우팅하고, 결과는 컴포넌트 레지스트리로 제약된 AUI로 보드/페이지에 렌더된다. 데이터가 쌓일수록 똑똑해진다.

## 1. 프로토타입 분석 & 인사이트

### 1.1 실제 빌드 화면에서 확인된 자산 (계승 · 스크린샷 기준)
- **마이보드(My Board)** = 자유형 크리에이티브 보드. 좌측 세로 **생성 툴바**, 멀티컬러 스티키, 콘텐츠/AI생성 카드, 주제 핀, 줌 컨트롤, **수업 모드** 버튼.
- **My Verse** = (별도 메뉴로 존재) 작업 캔버스. → **My Board로 통합(이 PRD 기준)**.
- **프롬프트바** = 채팅아이콘·+·입력·별(즐겨찾기/실행)·모드토글 + 상단 **즐겨찾기 카드**(놀이계획/놀이기록/관찰기록/문장생성/스튜디오).
- **AI 채팅** = "Claude 기반" **전용 페이지**(새 대화·추천 질문·최근 대화·추천 칩·면책 문구).
- 디자인은 이미 **크림 캔버스 + 코랄 악센트**(Milray Park) → 다크 제외 방침과 일치.
- 캔버스 인터랙션(팬·자유배치·줌·Fit·clip-path 확대) 계승.

### 1.2 프로토타입의 한계 (재설계에서 해결)
| 한계 | 해법 |
|---|---|
| 계획 없이 만들어 구조 일관성 부족 | §4 IA, §11 데이터 모델 |
| 에이전트 라우팅·계약 미정 | §5 3계층 + 출력/선택 계약 |
| UI 정적/생성형 경계 불명확 | §6 레지스트리 제약 AUI |
| 모델 비용·선택 전략 부재 | §7 라우팅·캐스케이드 |
| "똑똑해짐"이 느낌뿐 | §8 자가고도화 4-루프 |
| 아동 사진 거버넌스 부재 | §12 거버넌스 |
| My Verse/My Board 중복 | §4 통합 |

### 1.3 핵심 인사이트
1. 공간 + 자연어 + **선택(대상)** 이 정체성. 정적 셸과 생성형 결과를 분리.
2. 교사는 "끝내야 할 일"로 생각 → 의도 라우팅 + 즐겨찾기 카드.
3. 경쟁우위는 **누적 데이터 폐루프**.

### 1.4 ★ 교사 사용성 비판적 점검
| # | 리스크 | 대응 |
|---|---|---|
| R1 | 무한 캔버스 길 잃음 | 미니맵·Fit·홈·검색 등 **가장자리 컨트롤**(§4.3) |
| R2 | 빈 입력창 막막함 | **즐겨찾기 카드**(놀이계획 등) + 추천 질문 |
| R3 | AI가 관찰 지어냄 | 무근거 생성 금지 + 근거·영역 표시(§5.5/§12) |
| R4 | 결과 매번 다르게 보임 | 레지스트리 컴포넌트 형태 고정(§6) |
| R5 | 모바일/태블릿 | 태스크 우선 리스트 폴백(§6.3) |
| R6 | 지연감 | 스트리밍 + 낙관적 UI + 고속 모델 우선 |
| R7 | 콜드스타트 | 시작 템플릿/추천 질문(§8.3) |
| R8 | 과잉 자동화 불안 | 자율성 게이트 + "검토 후 내가 확정"(§9) |
| R9 | **컨트롤 과다 = 새 인지부하** | 항상 보이는 컨트롤 4종 최소화, 나머지는 **상황 시 등장**(§4.3) |

## 2. 제품 비전 & 목표

**비전**: "교사가 말하거나 고르기만 하면, 우리 원의 맥락을 아는 동료가 만들어 준다 — 확정은 언제나 교사가."

**1차 목표(6개월)**: 4대 태스크(놀이기록/관찰기록·놀이계획·통신문/문장·스튜디오) 자연어 생성+편집 / 라우팅 정확도 ≥90% / 수정 없이 채택률 30%→55%+.

**비목표(1차 제외)**: 킨더뷰 심층 통합 / 영상 풀기능(프리뷰 게이팅) / 모델 파인튜닝(RAG·메모리 대체).

## 3. 타깃 사용자 & 핵심 시나리오

**Primary persona** — **만 0~1·2·3·4·5세** 담임 교사. 영아반(0~2, 표준보육과정: 일상·기본생활습관 중심) / 유아반(3~5, 누리과정: 놀이중심). 종일 사용, 행정 부담, 모바일/태블릿 빈번.

**Job Stories**
1. 하원 후 사진 던지고 → "오늘 햇살반 블록놀이 관찰기록" → 연령 맞춤 초안 + 사진 자동 배치.
2. "다음 주 봄 주제 놀이계획" → 주안 카드.
3. "현장학습 가정통신문, 우리 원 톤" → 통신문 초안.
4. "봄꽃 색칠 도안 A4" → 스튜디오 이미지/도안.
5. (영아) "지호 이번 주 수유·낮잠 패턴 정리" → 영아 일상 기록.
6. 보드에서 카드 2~3개 선택 → "이거 묶어서 기록 한 장으로" → 병합 생성.

## 4. 정보구조(IA) & 화면 구조 (실제 빌드 반영)

### 4.1 LNB (좌측 아이콘 레일 — 실제 화면 기준)
```
├─ 홈            ← 랜딩/대시보드
├─ 갤러리        ← 사진·이미지·도안 자산 (분류·기억 엔진 결과)
├─ My Board      ← ★ 통합 캔버스 (구 My Verse 흡수)
├─ 우리반        ← 아동 등록·관리 (4.4)
├─ 캘린더        ← 일정 (계승)
├─ 폴더          ← 완성/진행 산출물 저장·정리
└─ 프로필(하단)  ← 계정·설정
        + AI 채팅 (상단/전역 진입 — 전용 페이지, 4.5)
```
> **My Verse 메뉴 완전 제거.** 별도 메뉴 없이 My Board로 단일화하며, 마이버스의 작업 캔버스 기능은 My Board 툴바의 '프레임/캔버스 추가'가 흡수(4.3). 기존 My Verse 데이터/보드는 마이그레이션으로 My Board에 편입.

### 4.2 My Board = 통합 캔버스 (핵심)
자유형 보드 + 작업 캔버스가 한 공간에 공존. "모으고·배치하고·완성한다."
- **자유 프리미티브**(툴바로 추가): 스티키 메모(멀티컬러), 텍스트, 이미지카드, 문서/메모 카드, 영상, 도형.
- **작업 캔버스/프레임**(구 My Verse): 놀이기록·놀이계획 등 *작업 자체*를 수행하는 캔버스를 보드에 얹음.
- **AI 생성 카드**: "선택 후 프롬프트로 AI 생성" — 선택한 대상/프롬프트로 결과 생성(§5.5).
- **워크플로 레인(★ 핵심)**: 한 번 요청하면 연관 자료·문서가 **가로로 한 레인 안에 순서대로 자라남**(§4.2.1).
- **주제 핀/존**: 보드 제목("우리 반 이야기")·주제 핀("이번 주 주제·가을")으로 구역화.

### 4.2.1 ★ Workflow Lane — 보드의 중심 상호작용
교사가 보드에서 **하나를 요청하면, 그와 연관된 자료/문서가 한 섹션(레인) 안에서 왼→오로 순서대로 펼쳐진다.** 채팅 로그가 아니라 *공간에서 작업 흐름이 자라는* 방식 — 킨더버스의 핵심 차별점.
- 예: "5월 가정의 달 활동 추천" → **놀이계획 레인** 생성 → ① 아이디어 리스트 →(복수선택)→ ② 활동 사진(AI 개념 이미지) → ③ 계획안(편집) → ④ 활동지(편집·저장) → **[레인 저장]→ 폴더에 번들**(계획안+활동지+이미지+연결자료).
- **템플릿 기본 + 라우터 동적 제안**: 단계 순서는 템플릿으로 고정(예측가능), 각 단계 후 라우터가 맥락 기반 `suggested_next`(예: "가정연계 통신문")를 *선택지*로 제안. **제안 레이어는 초기부터 탑재.**
- **진행은 클릭으로만**(자동 전체 실행 금지), **선택이 다음 단계 입력**, 모든 노드는 인라인+프롬프트바 편집.
- 구현은 **새 에이전트 없이 '워크플로 러너'** 한 겹이 기존 전문 에이전트(계획/스튜디오/문장/기록)를 템플릿 순서로 호출. 상세 명세·데이터 모델은 SKILL.md §9.
- 안전: ②의 활동 사진은 **AI 생성 개념 이미지**(실제 아동 사진과 분리, "AI 생성" 라벨). 발송·삭제는 자율성 게이트.

### 4.3 My Board 컨트롤 (스크린샷 기준 + R1/R9 보강)
**좌측 세로 툴바** (실제 화면):
| 도구 | 기능 |
|---|---|
| 선택(커서) | 대상 선택·복수 선택(=컨텍스트, §5.5) |
| 문서/페이지 | 문서·페이지 카드 추가 |
| 이미지 | 이미지카드 추가 |
| 텍스트(T) | 텍스트 블록 추가 |
| 메모/카드 | 스티키 메모·카드 추가 |
| 영상 | 영상 카드 추가 |
| 도형 | 도형/스티커 추가 |
| **프레임/캔버스** | **작업 캔버스(구 My Verse) 추가** ← 통합 핵심 |
| (하단) 잠금/홈 | 보드 잠금·홈 위치 복귀 |

**탐색/정리 컨트롤** (R1·R9 — 항상 보이는 건 최소화):
- 항상: **미니맵·Fit(맞춤)·검색·+추가**.
- 줌 컨트롤(우하단, —/82%/+/맞춤) · **수업 모드**(우상단, 보드를 교실 투사용 발표 뷰로).
- 상황 시 등장: **선택 툴바**(카드 선택 시 묶기·정렬·삭제·생성), 정렬/그리드 스냅, 존 만들기.

### 4.4 '우리반' 모듈
아동 등록·관리. 아동별 사진/문서 모아보기 + **특이사항·투약·출결석·하원시간**·메모. 영아/유아·반 편성. → §8 메모리·§5 슬롯의 1차 컨텍스트 소스.

### 4.5 'AI 채팅' 전용 페이지 (스크린샷 기준)
- 헤더: "AI 채팅 · Claude 기반".
- 좌측 패널: **+ 새 대화**, **추천 질문**(예: 가정통신문 초안·실내놀이 추천·관찰일지 누리과정 정리·안전교육 자료), **최근 대화**.
- 메인: "무엇을 도와드릴까요?" + 추천 칩 + 동일 프롬프트바.
- 푸터: **면책 문구**("AI는 실수할 수 있습니다 — 확인 필요").
- 역할: 자유 대화 + 새 작업 시작(어느 페이지든 열기). 작업 페이지 안에서는 프롬프트바가 그 페이지를 핸들링(§6.2).

### 4.6 갤러리 / 폴더 / 캘린더
갤러리=자산 라이브러리(분류 결과) · 폴더=산출물 저장·정리 · 캘린더=일정→생성 트리거 연계.

**상태 계층**: 전역(테넌트/교사) · 보드(위치/줌/배치/선택) · 태스크(진행 중 생성/편집). Zustand.

## 5. 에이전트 아키텍처 (핵심)

### 5.1 "얇은 3계층" + 공유 Pedagogy Foundation
```
        ┌───────────────────────────────────────────┐
        │ Pedagogy Foundation (공유 토대)             │
        │ 0~2 표준보육과정 · 3~5 누리과정 · 발달적합성 │
        │ 영역 연계 매핑 · 무근거 생성 금지            │
        └───────────────────────────────────────────┘
                ▲ 모든 Tier1 상속
[입력/선택]
   │
Tier 0 ── 라우터 — 빠른·저가 모델
   │  의도분류 / 슬롯추출 / **선택 컨텍스트** / 라우팅 / 확신도 (생성 X)
   ▼
Tier 1 ── 전문 에이전트 (기능명 통일, 적합성은 공유 레이어가 보장)
   ├─ 기록     : 관찰기록(영역분석·평가용) + 놀이기록=놀이이야기(사진배치·활동서술·학부모발송) (+ KB 오너)
   ├─ 계획     : 놀이계획 / 주안·월안 (활동 맥락을 스튜디오에 공급)
   ├─ 스튜디오 : 이미지·영상·도안 + 활동지/워크시트
   └─ 문장     : 문장생성 · 가정통신문·공지·평가서
   ▼
Tier 2 ── 결정형 도구
   ├─ 이미지/영상 생성 · 문서 템플릿 엔진
   └─ 분류·기억(엔진) : 사진 **실시간 분류**(기개발 API) + 임베딩·RAG
```
**즐겨찾기 카드 → 에이전트 매핑**: 놀이계획=계획 · 놀이기록/관찰기록=기록 · 문장생성=문장 · 스튜디오=스튜디오.
> 라우터는 시스템 내부 컴포넌트라 기능명("라우터")으로 통일. (내부 코드 식별자 예: `router`, `agent.record`, `agent.plan`, `agent.studio`, `agent.writing`, `engine.curator`.)

**산출물 분기·협업 (상세는 SKILL.md §1·§4.1)**
- **기록 2모드**: `observation`(관찰기록, 발달·영역분석, 행정/평가용) / `story`(놀이기록=놀이이야기·활동기록, 사진 배치+활동 서술, **학부모 발송용**). 놀이기록은 분류·기억 엔진이 사진 선별 → 기록이 서술, 부모 톤은 문장 에이전트 차용. 발송=L2/L3.
- **활동지**: 만드는 주체는 항상 스튜디오. (A) **놀이계획 연결** — 계획이 활동 맥락(연령·영역·목표)을 공급, 결과를 계획에 link. (B) **독립** — 라우터가 곧장 스튜디오로. 출력은 `WorksheetCard`(A4·인쇄·다운로드).

### 5.2 유아교육 적합성은 '공유 레이어'가 보장
계획·스튜디오·문장 포함 **모든 전문 에이전트가 Pedagogy Foundation을 상속**한다. '기록' 에이전트가 다른 에이전트를 직렬 검수하지 않는다(지연·병목·단일실패점 방지). '기록'은 지식베이스 *오너*. **고위험 산출물(평가서)에만** 출력 하네스의 **자동 적합성 검증 패스**(체크리스트) 1회.

### 5.3 에이전트 명명
**기능명으로 통일** — 라우터 / 기록 / 계획 / 스튜디오 / 문장 / 분류·기억. 교사 노출 라벨은 즐겨찾기 카드 그대로(놀이계획·놀이기록·관찰기록·문장생성·스튜디오), 내부 식별자는 위 기능명. (이전되는 핵심 가치는 라우터 + 2단계 패턴이며, 페르소나 이름은 사용하지 않는다.)

### 5.4 왜 3계층인가
단일콜=정확도·디버깅 악화 / 풀 멀티에이전트=지연·비용 폭증 / 3계층=라우팅·생성·도구 책임 분리.

### 5.5 ★ 출력 계약 + 선택=컨텍스트 + 안티-환각
프롬프트바는 **현재 페이지 + 선택 대상**을 항상 라우터에 동봉한다. "선택이 곧 범위."
```jsonc
{
  "page": "myboard",
  "selection": { "ids":["card_12","card_15"], "types":["image_card","memo"], "count":2 },
  "available_actions": ["generate","merge","relayout","restyle","add_media"],
  "intent": "merge",
  "scope": "selection",            // selection | page | new
  "route_to": "record",
  "lane": { "id":"lane_22","template":"play_plan","step":"plan","node_id":"node_3" },
  "suggested_next": [ { "action":"family_letter","label":"가정연계 통신문","reason":"5월 가정의 달","confidence":0.62 } ],
  "confidence": 0.93,
  "grounding": { "photos":["..."], "teacher_notes":["..."] }  // 없으면 생성 금지
}
```
**규칙**: ① 선택 있으면 그 대상에만 작용. ② `available_actions` 밖은 라우팅 안 함. ③ 레이아웃 변경=L1 자동(되돌리기), 생성·병합=미리보기 후 적용. ④ `confidence<0.7` 되묻기. ⑤ `grounding` 없으면 관찰 생성 금지·근거/영역 표시. ⑥ 다음액션 칩 = 기본 다음단계(템플릿, 항상) + `suggested_next`(라우터 제안 0~2, 옅게 "추천", 확신도<0.6 숨김). 제안은 선택지일 뿐 자동 실행 금지.

### 5.6 MeetFlow 로스터 대비
MeetFlow 패턴 대비: 승계=라우터·2단계. 신규=분류·기억 엔진, Pedagogy Foundation. 페르소나 이름(프뢰벨 등)은 폐기하고 기능명으로 통일.

## 6. AUI & 상호작용 전략

### 6.1 경계
| 영역 | 방식 |
|---|---|
| 셸·LNB·My Board 캔버스·툴바·갤러리·우리반·캘린더·폴더·AI채팅 셸 | **정적 UI** |
| 결과 표면(기록·계획·통신문·이미지 갤러리·AI생성 카드) | **AUI** |

### 6.2 ★ 프롬프트바 = 컨텍스트 인식 핸들러
하나의 바, **페이지마다 다른 능력**. 각 페이지가 받을 수 있는 `available_actions`를 등록 → 라우터는 그 범위에서만 라우팅.
- **AI 채팅 페이지**: 일반 대화 + 새 작업 시작.
- **작업 페이지(놀이기록 등)**: 그 작업물 핸들러(사진 추가·문단 수정·영역 재연계·톤 변경).
- **My Board**: 보드 핸들러(카드 추가·선택 대상 정렬/묶기/생성).
- **즐겨찾기 카드**(놀이계획/놀이기록/관찰기록/문장생성/스튜디오): 입력이 빈 상태에서 **별 버튼**을 누르면 바 위로 솟아오르고, 카드 클릭 시 해당 페이지로 이동. **추천 칩은 선택 상태에 따라 동적 변경**(카드 2개 선택 시 "묶기/한 장으로" 등). 좌측 **메시지 아이콘=AI 채팅 이동**, 우측 **둥근 토글=바 접기/펼치기**. 상세 동작·구현 기준은 §10.4.

### 6.3 AUI = 레지스트리 선택 + 파라미터 (임의 생성 금지)
`에이전트 JSON → 스키마 검증 → UI Registry 매칭 → 정적 컴포넌트 렌더`
**Registry 카탈로그(초기)**: `RecordDraftCard`(관찰기록)·`PlayStoryCard`(놀이기록=놀이이야기, 사진슬롯+활동서술+학부모발송)·`WeeklyPlanGrid`·`LetterPreview`·`StudioGallery`(이미지/영상/도안)·`WorksheetCard`(활동지/워크시트, A4·인쇄·다운로드·계획연결)·`PhotoClusterBoard`·`StickyNote`·`TextBlock`·`ImageCard`·`WorkCanvasFrame`(구 My Verse)·`ClarifyPrompt`·`FavoriteCardRail`·`SelectionToolbar`.

### 6.4 반응형 (모든 뷰포트 + 모바일 폴백)
셸=container query(LNB 레일↔하단탭) · 결과=fluid grid · **My Board=무한 캔버스+줌/팬, 작은 화면은 태스크 우선 리스트로 폴백(R5)** · BP: sm480/md768/lg1024/xl1440/2xl1920+.

| **좌측 메시지 아이콘** | 어느 페이지에서든 클릭 시 **AI 채팅 페이지로 즉시 이동**(전역 바로가기). 또한 바가 닫힌 상태에서는 이 버튼만 남아 **바를 다시 펼치는 트리거**. |
| **+ 추가** | 첨부/대상 추가. |
| **입력창** | "무엇이든 물어보세요". 현재 페이지 + 선택 대상을 컨텍스트로 동봉(§5.5). |
| **별 ↔ 전송(토글)** | **입력이 비어 있으면 = 별(즐겨찾기) 버튼**, **입력이 있으면 = 전송(게시) 버튼**으로 전환. |
| **별 클릭 (입력 없음)** | 바 위로 **즐겨찾기 카드 솟아오름**(놀이계획·놀이기록·관찰기록·문장생성·스튜디오). 카드 클릭 → **해당 페이지로 이동**. |
| **우측 끝 둥근 토글** | 클릭 시 바가 **접혀 메시지 아이콘 버튼 하나만 남음**. 다시 그 버튼으로 펼침. |

스트리밍 중 인라인 진행 표시. 추천(즐겨찾기) 카드는 선택 상태에 따라 동적 변경(§6.2).

> ⚙️ **개발 지침**: 위 프롬프트바 동작(메시지 아이콘→AI채팅 이동 / 둥근 토글→바 접기·펼치기 / 별↔전송 토글 / 별→즐겨찾기 카드→페이지 이동)은 **새로 구현하지 말고 `localhost:5173` 프로젝트의 기존 프롬프트바 컴포넌트 코드를 기준 구현체로 가져와 그대로 적용**한다. 정식 서비스에서는 이 컴포넌트를 공용 셸 컴포넌트로 승격해 전 페이지에서 재사용한다.
