# My Board 성능 베이스라인

> My Board(무한 캔버스) 성능 개선 작업의 측정 기록. 각 Phase 2 개선 항목은
> 아래 동일 절차로 **전/후**를 측정해 이 문서에 누적 기록한다.
> 측정 도구: `src/dev/perfTools.ts` (DEV 전용, 프로덕션 빌드 제외).

## 측정 환경

| 항목 | 값 |
|---|---|
| 측정 보드 | `__kvSeedPerf(500)` — 이미지 250 + 스티키/텍스트/도형 250, 4000×3000 분산 |
| 브라우저 | Chromium (preview 하니스) |
| 빌드 모드 | dev (`npm run dev`, Vite) |
| React | StrictMode (dev 이중 렌더 — 절대값보다 **개선 전/후 상대비**로 해석) |

> ⚠️ dev + StrictMode라 절대 FPS는 프로덕션보다 비관적이다. 개선 효과는
> **동일 환경 전/후 비율**로 판단한다(컬링·memo 적용 후 같은 절차로 재측정).

## 측정 절차 (재현 가능)

1. `npm run dev` 후 `http://localhost:5173/board?fps=1` 접속 (FPS 오버레이 자동 ON).
2. 브라우저 콘솔에서 `__kvSeedPerf(500)` 실행 → "⚡ Perf 500" 보드로 전환됨.
3. 아래 4개 지표를 측정한다. FPS는 오버레이의 `FPS` 값(직전 0.5초 평균)과 `min`(측정 구간 최저)을 함께 기록.
   - **(a) 초기 로드**: `__kvSeedPerf(500)` 로그의 `seeded … in Nms` + 시드 직후 화면이 응답 가능해질 때까지 체감 시간. 보조로 DevTools Performance 패널의 스크립팅/렌더 시간.
   - **(b) 팬 FPS**: 빈 영역을 잡고 5초간 빠르게 드래그-팬하며 오버레이 FPS 관찰(최저값 기록).
   - **(c) 줌 FPS**: 마우스 휠로 5초간 연속 줌 인/아웃하며 관찰.
   - **(d) 카드 드래그 FPS**: 카드 1개를 잡고 5초간 화면을 가로질러 이동하며 관찰.
4. 각 측정 3회 반복 후 중앙값 기록.

> ⚠️ 팬/줌/드래그는 수동 제스처라 사람이 직접 측정해야 한다. 자동화가 어려운
> 지표는 "수동 측정 필요"로 표기하고, 측정자가 위 절차로 채운다.

## 네트워크(이미지 전송량) 측정 — DoD용

> 주의: 현재 이미지는 **base64 data URI**로 인라인 저장되어 네트워크 전송이
> 발생하지 않는다. 따라서 DoD의 "네트워크 탭 기준 이미지 전송량"은 현 구조에
> 직접 적용되지 않는다. Phase 2-1(이미지 LOD, 클라이언트 썸네일) 적용 후에는
> **메모리·디코드 비용 / DOM 이미지 픽셀 총량**으로 대체 측정한다(절차는 2-1에서 확정).

## 베이스라인 (개선 전) — Phase 1

측정일: 2026-06-10 · 방식: preview 하니스 자동 측정(viewport 구동 rAF 벤치)

| 지표 | 측정값 | 비고 |
|---|---|---|
| (a) 시드 빌드 시간 | **861ms** | 더미 250개 JPEG 생성 포함 — 도구 오버헤드. 실제 보드 로드 아님 |
| (a') DOM 렌더 결과 | 노드 500 / `<img>` 250 전부 DOM 존재 | **컬링 없음 확정** (뷰포트 밖도 전부 렌더) |
| (b) 팬 FPS (avg / min) | **30 / 15** | 최악 프레임 65ms. 자동 팬 벤치(2.5s, viewport 진동) |
| (c) 줌 FPS (avg / min) | ≈ 팬과 동일 (미실측) | 줌도 viewport-only 경로라 동일 비용 — 수동 확인 권장 |
| (d) 카드 드래그 FPS (avg / min) | _수동 측정 필요_ | 포인터 제스처 필요(자동화 부정확). 절차대로 측정 |

**해석**: 500개 보드에서 팬 평균 30fps / 최저 15fps로 **목표(55fps) 미달**. 원인은
진단대로 ⓐ 컬링 부재(500개 상시 렌더) + ⓑ `React.memo` 부재(viewport 변경 매
프레임 500개 NodeView 본문 재실행). Phase 2-1(컬링)·2-3(memo)가 직접 타깃.

> (c) 줌·(d) 드래그는 다음 측정 시 위 절차로 채운다. (b) 팬이 가장 대표적인
> 핫패스라 자동 벤치로 우선 확보했다.

## 개선 항목별 전/후 기록 (Phase 2)

각 항목 적용 시 아래 표를 복제해 채운다.

### 2-1. 뷰포트 컬링 ✅
측정: 500노드 보드, **줌 100%**(가시 84/507 — fit 28%는 전부 보여 컬링 효과 미노출이라 줌1로 측정).

| 지표 | 전(컬링 없음) | 후(컬링) | Δ |
|---|---|---|---|
| 팬 FPS (avg/min) | 30 / 14 | **46 / 18** | **+53% / +29%** |
| 최악 프레임(ms) | 70 | 55 | −21% |
| 렌더된 이미지 카드(`<img>`) | 250 (전부) | **154** | −38% |
| 렌더된 노드(추정) | 500 (전부) | ~300 | −40% |

구현: `BoardCanvas` — 뷰포트 가시 rect를 카메라(x,y,zoom)+캔버스 치수로 계산, **50% 마진** 확장 후 각 노드 AABB 교차만 렌더. keep-set = 선택 ∪ 드래그 중 ∪ 마인드맵 엣지 끝점(편집 카드 언마운트·연결선 끊김 방지). 시각/인터랙션 회귀 없음(스크린샷 확인). 줌이 깊을수록(가시 노드 적을수록) 효과 더 큼. 목표 55fps에는 2-3(memo)와 합산해 도달 예정.

### 2-2. 이미지 LOD (클라이언트 썸네일 + 저줌 강등) ✅ (결정 A)
측정: 500노드 보드(이미지 250장 — 시드를 실제 게이트웨이 해상도인 **1024×768**로 상향), 줌 100%.

| 지표 | 전(원본 직접 표시) | 후(썸네일 표시) | Δ |
|---|---|---|---|
| 표시 이미지 해상도 | 1024×768 | **400×300** | — |
| 디코드 픽셀 총량(가시 154장) | 121.1M px | **18.5M px** | **−85%** |
| 줌<30%에서 `<img>` 수 | 154 | **0** (플레이스홀더) | −100% |
| 팬 FPS (avg/min) | — | 60 / 45 | FPS 유지(원본 2.6배 커졌는데도) |

구현: `src/board/imageLod.ts` — `ensureThumb()`이 이미지 카드의 원본(data URI)을
`<canvas>`로 ≤400px JPEG 썸네일로 축소해 `node.data.thumb`에 저장(동시 4개 제한 큐,
NodeView 마운트 시 lazy — 컬링 덕에 보인 카드만). 보드 `<img>`는 `thumb || src`로
표시하고 **원본 `node.src`는 그대로 보존**(확대/편집/내보내기/저장 경로 무변경).
줌 < `IMG_PLACEHOLDER_ZOOM`(0.3)이면 BoardCanvas가 `lod` prop을 내려 이미지를
단색 플레이스홀더+제목으로 강등(디코드 0). boolean prop이라 memo 효과 유지.

> ✅ **DoD 대체 지표 충족**: "이미지 전송량 70%↓"는 data URI 인라인 구조라 네트워크
> 전송이 없어 **디코드 픽셀 총량으로 대체** — −85% ≥ 70% 달성.
> 부가 효과: 2-4 localStorage 쿼터 부담도 표시 경로와 무관하게 동일(원본 보존 정책).
> 시드 해상도 상향으로 이전 FPS 표와 절대 비교는 불가(동일 조건 내 비교만 유효).

### 2-3. memo (NodeView) ✅
측정: 500노드 보드, 줌 100%. "전"=컬링만(2-1), "후"=컬링+memo.

| 지표 | 전(컬링만) | 후(컬링+memo) | Δ |
|---|---|---|---|
| 팬 FPS (avg/min) | 46 / 18 | **60 / 43** | **+30% / +139%** |
| 최악 프레임(ms) | 55 | 23 | −58% |

구현: `BoardCanvas`에서 `MemoNodeView = memo(NodeView)`로 래핑 + `onPointerDown`을
latest-ref 패턴으로 **안정 콜백**화(매 렌더 새 핸들러가 memo를 무력화하던 문제 제거).
팬/줌 시 카메라만 바뀌고 각 카드의 props(node/selected/dx/dy/onPointerDown)는 불변 →
memo가 렌더된 ~300개 NodeView 본문 재실행을 전부 건너뜀. **NodeView.tsx는 세션 시작
전 무관 변경이 있어 건드리지 않음** → memo 래핑을 BoardCanvas 측에서 처리.

> ✅ **DoD 충족**: 팬 평균 60 / 최저 43 fps ≥ 55fps 목표 (500노드 @ 줌100%, dev+StrictMode).
> 컬링 단독 46fps → memo 합산 60fps로 달성.
>
> 미적용(보류): 카드 `contain: layout paint` / 드래그 `will-change`는 NodeView.tsx 직접
> 수정이 필요한데 해당 파일에 기존 미커밋 변경이 섞여 있어, 분리 커밋을 위해 보류.
> 현재 FPS가 목표를 넘겨 필수는 아님(추후 NodeView 변경이 커밋되면 추가).

### 2-4. localStorage 디바운스 영속화 ✅ (결정 C)
| 지표 | 전 | 후 | 비고 |
|---|---|---|---|
| 새로고침 후 복원 | **불가**(인메모리) | **복원됨** | 검증: 카드 2개 추가→리로드→텍스트 그대로 복원 |
| 저장 트리거 | 보드 전환/언마운트 1회 | 라이브 편집마다 **800ms 디바운스** + 탭 종료 flush | |

구현: `src/board/persist.ts` — `initBoardPersistence()`를 `main.tsx`에서 렌더 전 호출.
부팅 시 localStorage(`kv:boards:v1`)에서 보드 목록+스냅샷+activeId를 하이드레이트(복원된
activeId가 MyBoardPage의 "보드 1개 보장" 이펙트보다 먼저 자리잡아 중복 생성 없음).
이후 `useBoardStore` 변경을 구독해 800ms 디바운스로 `saveActiveLive()`→전체 스냅샷을
localStorage에 기록. **스키마 변경 없음**(boardsStore/boardStore 미수정, `setState`로 주입).

> ⚠️ **알려진 한계**: 이미지가 base64 data URI라 이미지 많은 보드는 localStorage 쿼터
> (~5MB) 초과 가능 → `QuotaExceededError`는 catch 후 경고(1회 재시도), 보드는 유지되되
> 그 사이클 미저장. **2-2(이미지 LOD 썸네일)** 적용 시 페이로드가 크게 줄어 해소됨.
