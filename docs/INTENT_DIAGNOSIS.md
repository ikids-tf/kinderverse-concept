# 의도 파악 진단 + 보강 계획

> 증상: ① 교사가 교육 현장 어휘로 입력하면 의도를 잘 못 알아듣는다.
> ② 화면(보드) 위 대상에 지시할 때도 의도 파악이 안 된다.
> 본 문서는 코드 근거 기반 원인 진단과 단계별 보강 계획이다. (분석 시점: 2026-06-10)

---

## 1. 의도 해석이 일어나는 3개 층 (현재 구조)

```
교사 입력 (프롬프트바)
   │
   ├─ [층1] 보드에서 선택이 있을 때 → src/board/prompt.ts handleBoardPrompt()
   │        = 100% 정규식 키워드 매칭 (모델 안 거침)
   │
   ├─ [층2] 선택이 없을 때 → composer.ts composeFromPrompt() → Tier-0 라우터
   │        = 모델 호출이지만, dev(키 없음)에선 mock 키워드 30개 룰
   │        = 모델이 답해도 후단 정규식이 덮어쓰는 지점 존재
   │
   └─ [층3] 화면 조작 지시("크게/옮겨/정렬/지워") → 실행 경로 자체가 없음
```

## 2. 원인 진단 (근거 파일·라인)

### 원인 A — 의도 결정의 대부분이 "작은 정규식 사전" (층1)
- `src/board/prompt.ts` `detectIntent()`: 활동지/계획안/통신문/이미지 정규식 4개,
  표면형 합계 **약 40개 단어**가 전부. 매칭 실패 시 **기본값 `'text'`**.
- 결과: 교사 어휘가 사전에 없으면 — 예: "**동시** 지어줘", "**신체표현** 활동",
  "**이야기나누기** 자료", "**환경판** 꾸밀 그림", "**알림장** 써줘", "**명렬표**",
  "**융판동화**", "**손유희**" — 전부 `'text'`로 오인되어 **선택한 메모를 조용히
  덮어쓰거나** 이미지를 엉뚱하게 재생성한다. *오해가 질문 없이 실행되는 것*이
  체감 "의도 파악 못함"의 핵심.

### 원인 B — 어휘 사전이 3곳에 따로 존재(불일치)
- `src/board/prompt.ts` (WORKSHEET_RE·PLAN_RE·LETTER_RE·IMAGE_RE·DESIGN_CMD_RE·DECORATE_RE)
- `src/board/composer.ts` (COLORING_RE·MEDIA_RE·WORKSHEET_REQ_RE·MINDMAP_RE — 53~59행)
- `server/gateway/mock.ts` RULES (22~28행, 키워드 ~30개)
- 같은 의도인데 층마다 인식 어휘가 다름. 예: '포스터·움짤·gif'는 composer의
  MEDIA_RE에만 있고 prompt.ts IMAGE_RE에는 없음 → 선택 유무에 따라 같은 문장이
  다르게 해석된다.

### 원인 C — dev에서는 "AI 라우터"가 사실상 없음 (층2)
- API 키 미설정 시 라우터는 `mock.ts mockRouterOutput()` = **키워드 30개 룰**.
  ("비 오는 날 실내에서 에너지 발산할 만한 거" → 키워드 0개 → 무조건 clarify)
- **현재 키 설정 여부 확인 필요** — 키 없이 시연 중이었다면 체감 품질의 상당
  부분이 이 mock 한계다. (라우터 응답의 `mocked: true` 플래그로 즉시 판별 가능)
- mock의 SELECTION_RULES(30~34행)는 매칭 시 `route_to: 'record'`로 보낸다 —
  병합/정렬 요청이 기록 에이전트로 가는 **버그성 매핑**.

### 원인 D — 선택 컨텍스트가 라우터에 안 들어감
- `PromptBar.tsx:180`, `composer.ts:76` 모두 `selection: { ids, types: [], count }`
  — **types가 항상 빈 배열**. 선택한 게 이미지인지 문서인지, 무슨 내용인지
  모델이 전혀 모른다. "선택이 곧 범위"(SKILL §3)인데 선택의 *내용*이 누락.

### 원인 E — 모델이 잘 분류해도 정규식이 덮어씀
- `composer.ts:84` `if (out.route_to === 'mindmap' || MINDMAP_RE.test(text))`,
  `effectiveAgent()`(720~730행)의 COLORING/WORKSHEET/MEDIA 분기 — 라우터 판단과
  무관하게 정규식이 우선하는 지점들. 라우터를 개선해도 효과가 차단됨.

### 원인 F — 화면 조작 의도의 "실행 경로" 부재 (층3)
- "이거 더 크게", "왼쪽으로", "나란히 정렬", "지워줘", "복사", "노란색으로" →
  보드 store에는 raw op(moveNodesRaw 등)가 있지만 **프롬프트→조작 연결이 없음**.
- `PAGE_ACTIONS['/board']`의 merge/relayout/restyle은 선언만 있고 실행기 없음.
- 유일한 경로는 "프레임 선택 + DESIGN_CMD_RE" → redesignFrame 뿐.

## 3. 보강 계획

### P0 — 어휘·컨텍스트 (작고 빠른 효과, 코드 변경 소)
| # | 항목 | 내용 | 대상 파일 |
|---|---|---|---|
| 1 | **의도 어휘 사전 단일화** | `src/ai/intent-lexicon.ts` 신설 — 의도별 표면형을 한 곳에. 유아교육 어휘 확장: 누리 5영역 활동어(신체표현·동시·동화·이야기나누기·바깥놀이·역할/쌓기놀이·미술·음률·요리활동·손유희·융판), 행사/서류(알림장·명렬표·출석부·이름표·상장·초대장·환경판·게시판·주간안내), 조작 동사(키워/줄여/옮겨/정렬/묶어/지워/복사/색). prompt.ts·composer.ts·mock.ts가 전부 이걸 import | 신규 + 3파일 치환 |
| 2 | **selection.types 실전달** | `types: []` 하드코딩 제거 — 실제 노드 타입 + role + 제목 1줄 요약을 라우터 입력에 포함 | PromptBar.tsx, composer.ts |
| 3 | **mock 라우터 강화** | 사전(1) 기반으로 재작성 + SELECTION_RULES의 record 오라우팅 수정 | mock.ts |

### P1 — 라우팅 구조 (모델을 의도 판단의 주인으로)
| # | 항목 | 내용 |
|---|---|---|
| 4 | **선택 경로에도 라우터** | prompt.ts: 정규식은 "고확신 fast-path"로 유지하되, 미매칭 시 `'text'` 기본값으로 조용히 실행하지 말고 **Tier-0 라우터 호출** → 그래도 모호하면 기존 불일치 팝업(PromptChoiceDialog)/clarify 재사용. "오해하면 묻는다" 원칙 |
| 5 | **라우터 few-shot** | L2_TASK에 교사 실발화 예시 8~12개(입력→route_to/intent JSON) + 어휘→에이전트 매핑 표 추가. 토큰 ~600 증가, low-tier로 충분 | 
| 6 | **정규식 오버라이드 정리** | composer의 MINDMAP/effectiveAgent 분기: 라우터 confidence ≥ 0.7이면 라우터 판단 존중, 정규식은 라우터 실패 시 폴백으로 강등 |

### P2 — 화면 지시(보드 조작) 신설
| # | 항목 | 내용 |
|---|---|---|
| 7 | **board_action 계약 + 실행기** | RouterOutput에 `board_action?: { type: 'move'\|'resize'\|'align'\|'arrange'\|'delete'\|'duplicate'\|'recolor', params }` 추가. `src/board/actions.ts` 실행기 신설 — 기존 raw ops + commands(undo) 재사용. L1 자동(되돌리기 가능), delete만 확인 |
| 8 | **PAGE_ACTIONS 정합** | '/board'에 위 액션 등록, merge/relayout/restyle 실행기 연결(또는 제거) |

### P3 — 품질 루프 (회귀 방지·지속 개선)
| # | 항목 | 내용 |
|---|---|---|
| 9 | **골든셋 + 측정 스크립트** | 교사 발화 50~100문장(현장 어휘 변형 포함) → 기대 라우팅 CSV. dev 스크립트로 mock/실모델 정확도 리포트 — 사전·프롬프트 수정 시 회귀 즉시 탐지 |
| 10 | **정정 UX** | 생성 결과 카드에 "다른 걸 원하셨나요?" 칩 → 원클릭 재라우팅. 정정 로그를 골든셋에 축적 |

### 권장 순서와 기대 효과
1. **P0(1~3)**: 어휘 누락으로 인한 오인식 즉시 감소 — 반나절 작업.
2. **P1(4~6)**: "조용한 오실행" 제거 + 모델 이해력 활용 — 1일.
3. **P2(7~8)**: 화면 지시가 비로소 동작 — 1~1.5일.
4. **P3(9~10)**: 이후 모든 개선의 측정 기반 — 0.5일.
