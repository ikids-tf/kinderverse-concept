## 1. 에이전트 로스터 (3계층, 기능명)
| 계층 | 식별자 | 역할 | 모델 |
|---|---|---|---|
| Tier0 | `router` | 의도분류·슬롯추출·선택 컨텍스트·라우팅·확신도 (생성 X) | 저가·고속 |
| Tier1 | `agent.record` | **2모드**: `record.observation`(관찰기록—발달·누리/표준 영역분석, 행정/평가용) · `record.story`(놀이기록=놀이이야기·활동기록—사진 배치+활동 서술, **학부모 발송용**). KB 오너 | 중급 |
| Tier1 | `agent.plan` | 놀이계획·주안·월안 (활동 맥락을 스튜디오에 공급) | 중급 |
| Tier1 | `agent.studio` | 이미지·영상·도안 + **활동지/워크시트**(프롬프트 설계 + 도구 오케스트레이션) | 이미지/영상 전용 |
| Tier1 | `agent.writing` | 문장생성·가정통신문·공지·평가서 | 중급 |
| Tier2 | `engine.curator` | 사진 실시간 분류(기개발 API) + 임베딩·RAG·기억 | (엔진) |

즐겨찾기 카드 → 에이전트: 놀이계획=plan · 놀이기록/관찰기록=record · 문장생성=writing · 스튜디오=studio.

## 2. Pedagogy Foundation (공유 토대 레이어 — 모든 Tier1 상속)
- **0~2세 = 표준보육과정** (일상·기본생활습관·정서적 상호작용 중심).
- **3~5세 = 누리과정** (놀이중심·자율탐구, 5개 영역 연계).
- 발달 적합성 규칙, 영역 연계 매핑, **무근거 생성 금지** 규칙 포함.
- 적합성은 이 레이어가 보장한다. `agent.record`는 KB 오너일 뿐 다른 에이전트를 직렬 검수하지 않는다.
- 고위험 산출물(발달평가서)만 출력 하네스에서 **자동 적합성 검증 패스(체크리스트)** 1회.

## 3. 출력 계약 (Output Contract) — 자유 텍스트 금지, JSON만
라우터는 항상 **현재 페이지 + 선택 대상**을 동봉한다. "선택이 곧 범위."
```jsonc
{
  "page": "myboard",                               // 현재 페이지
  "selection": { "ids":["card_12","card_15"], "types":["image_card","memo"], "count":2 },
  "available_actions": ["generate","merge","relayout","restyle","add_media"],
  "intent": "merge",
  "scope": "selection",                            // selection | page | new
  "route_to": "record",                            // router|record|plan|studio|writing
  "mode": "story",                                 // record: observation|story / 그 외 생략
  "link": { "plan_id": null, "child_ids": [] },    // 활동지↔놀이계획, 산출물↔아동 연결(선택)
  "lane": { "id":"lane_22", "template":"play_plan", "step":"plan", "node_id":"node_3" }, // 워크플로 레인 컨텍스트(§9)
  "suggested_next": [                              // 라우터의 상황 제안(0~2). 기본 다음단계와 별도
    { "action":"make_worksheet", "label":"활동지 만들기", "reason":"계획안 완료", "confidence":0.9 },
    { "action":"family_letter", "label":"가정연계 통신문", "reason":"5월 가정의 달 맥락", "confidence":0.62 }
  ],
  "confidence": 0.93,
  "grounding": { "photos":["..."], "teacher_notes":["..."] }  // 없으면 관찰/활동 서술 생성 금지
}
```
**규칙**
1. 선택이 있으면 그 대상에만 작용(페이지 전체 미변경).
2. `available_actions` 밖은 라우팅하지 않는다.
3. 레이아웃 변경=L1 자동(되돌리기). 생성·병합=미리보기 후 적용.
4. `confidence < 0.7` → 명확화 질문(추측 금지).
5. `grounding` 비면 관찰/활동 서술 생성 금지. 결과에 근거 출처 + 연계 영역 표시.
6. **학부모 발송 산출물**(놀이기록·통신문)은 생성=L1 초안, **발송=L2 확인 / 외부 채널 발송=L3**.
7. **다음액션 칩 = 기본 다음단계(템플릿, 항상) + `suggested_next`(라우터 제안, 0~2개)**. 시각 구분: 기본=진하게, 제안=옅게 "추천" 태그. 제안 확신도<0.6이면 숨김. 제안은 *선택지*일 뿐, 자동 실행 금지(교사 클릭으로만 진행).

## 4. UI Registry (AUI는 레지스트리 선택+파라미터만)
`에이전트 JSON → 스키마 검증 → Registry 매칭 → 정적 컴포넌트 렌더`
초기 카탈로그: `RecordDraftCard`(관찰기록), `PlayStoryCard`(놀이기록=놀이이야기—사진 슬롯 그리드 + 활동 서술 + 학부모 발송 미리보기), `WeeklyPlanGrid`, `LetterPreview`, `StudioGallery`(이미지/영상/도안), `WorksheetCard`(활동지/워크시트—A4·인쇄 규격·다운로드, 연결된 놀이계획 표시), `PhotoClusterBoard`, `StickyNote`, `TextBlock`, `ImageCard`, `WorkCanvasFrame`(보드에 얹는 작업 캔버스), `ClarifyPrompt`, `FavoriteCardRail`, `SelectionToolbar`.
각 컴포넌트의 props 스키마 = 에이전트 출력 계약과 1:1.

### 4.1 에이전트 협업 경로 (단일 책임 + 컨텍스트 공급)
- **놀이기록(`record.story`)**: `engine.curator`가 그날 활동 사진을 골라줌 → `record.story`가 사진에 맞춰 활동 서술 생성 → `PlayStoryCard` 렌더. 부모 발송 톤은 `agent.writing`의 톤 능력을 선택적으로 빌림.
- **활동지(워크시트/도안)**: 만드는 주체는 항상 `agent.studio`. 단 두 경로 —
  - **경로 A(계획 연결)**: `agent.plan`의 특정 활동에서 "이 활동 활동지 만들기" → 계획이 활동 맥락(연령·영역·목표)을 `agent.studio`에 공급 → 결과를 `link.plan_id`로 연결.
  - **경로 B(독립)**: 라우터가 곧장 `agent.studio`로. 연령·영역 슬롯만 받아 Pedagogy Foundation에 맞게 생성.
  - 두 경로 모두 출력은 `WorksheetCard`. 활동지↔놀이계획 연결은 폴더에서 묶여 표시.

## 5. 정보구조 / LNB
상단: 홈 · 갤러리 · **My Board**(통합 캔버스) / 중단: 우리반 · 캘린더 · 폴더 / 하단: 프로필. 전역: **AI 채팅(전용 페이지)**.
- **우리반**: 아동 등록·관리(아동별 사진/문서 + 특이사항·투약·출결·하원시간). → 메모리/슬롯의 1차 컨텍스트.
- **갤러리**: 자산 라이브러리(분류·기억 엔진 결과). **폴더**: 산출물 저장·정리. **캘린더**: 일정→생성 트리거.

## 6. My Board (통합 캔버스) 명세
자유 프리미티브(스티키·텍스트·이미지카드·문서/메모·영상·도형) + 작업 캔버스(프레임 추가) + AI 생성 카드 + 주제 핀/존.
**좌측 세로 툴바**: 선택(커서) / 문서 / 이미지 / 텍스트(T) / 메모·카드 / 영상 / 도형 / **프레임·캔버스 추가**(구 My Verse 흡수) / (하단)잠금·홈.
**탐색·정리 컨트롤** — 항상: 미니맵·Fit·검색·+추가 / 우하단 줌(—/%/+/맞춤)·우상단 수업 모드(교실 투사 뷰) / 상황 시 등장: 선택 툴바(묶기·정렬·삭제·생성), 정렬/그리드 스냅, 존 만들기.
※ 컨트롤 과다는 인지부하(R9) → 항상 보이는 건 최소화, 나머지는 컨텍스트에 따라 등장.

### 6.1 키보드 단축키 (전 작동 필수)
표준 보드 단축키를 모두 지원한다. (Mac=Cmd, Win=Ctrl)
| 동작 | 단축키 |
|---|---|
| 실행취소 / 다시실행 | `⌘/Ctrl+Z` / `⌘/Ctrl+Shift+Z`(또는 `Ctrl+Y`) |
| 삭제 | `Delete` / `Backspace` |
| 복수선택 | `Shift+클릭`(누적) · 빈공간 **드래그 박스 선택** · 전체선택 `⌘/Ctrl+A` |
| 복사 / 잘라내기 / 붙여넣기 | `⌘/Ctrl+C` / `X` / `V` |
| 복제 | `⌘/Ctrl+D` |
| 그룹 / 해제 | `⌘/Ctrl+G` / `⌘/Ctrl+Shift+G` |
| 줌 인/아웃/리셋 | `⌘/Ctrl+=` / `⌘/Ctrl+-` / `⌘/Ctrl+0` |
| 전체 맞춤(Fit) | `Shift+1` (빈공간 더블클릭도 Fit) |
| 팬(이동) | `Space+드래그` |
| 선택 해제 | `Esc` |
| 잠금/해제 | `⌘/Ctrl+L` |
- 입력창/텍스트 편집 중에는 보드 단축키를 가로채지 않는다(포커스 컨텍스트 분리).
- 모든 단축키에는 대응하는 마우스/버튼 경로가 함께 존재해야 한다(키보드 없이도 가능).
- **기존 로컬 컴포넌트(예: 프롬프트바)에 이미 구현된 단축키 동작은 보존**하고 충돌만 정리한다.

### 6.2 Undo/Redo 히스토리 (설계 요구)
단축키만으로는 부족 — **명령 히스토리 스택**이 받쳐야 실제로 작동한다.
- 모든 보드 동작(추가·이동·리사이즈·삭제·그룹·레이아웃 변경·**AI 생성/병합 적용**)을 **되돌릴 수 있는 단위(command)** 로 기록.
- undo/redo 스택은 보드 상태(Zustand)와 분리된 히스토리 모듈로. 페이지 이탈/새로고침 시 동작 정의(세션 한정 or 영속) 명시.
- 자율성 게이트와 연결: **L1(생성 초안·분류·레이아웃)=undo 가능**.
- **비가역/고위험 동작은 단순 Ctrl+Z 대상이 아님**: AI 생성·병합은 "미리보기 → 적용"이라 **적용 취소(undo)** 로 되돌리되, **외부 발송·영구 삭제(L3)** 는 단축키로 즉시 실행되지 않게 막고 확인 모달 + (가능 시) 일정 시간 '실행취소' 토스트로 처리.

## 7. 프롬프트바 (컨텍스트 핸들러 — 기존 로컬 컴포넌트 재사용)
좌→우: **메시지 아이콘 · + · 입력창 · 별/전송 · 둥근 토글**.
- 좌측 메시지 아이콘: 어느 페이지에서든 → **AI 채팅 페이지로 이동**. 바가 닫힌 상태에선 이 버튼이 **펼침 트리거**.
- 별 ↔ 전송: 입력 비면 **별(즐겨찾기)**, 입력 있으면 **전송**.
- 별 클릭(입력 없음): 바 위로 **즐겨찾기 카드 솟아오름**(놀이계획·놀이기록·관찰기록·문장생성·스튜디오) → 클릭 시 해당 페이지 이동.
- 우측 둥근 토글: 바를 접어 **메시지 아이콘만** 남김(재클릭 펼침).
- 페이지별로 `available_actions`를 등록 → 라우터는 그 범위에서만 라우팅. 추천 칩은 선택 상태에 따라 동적 변경.
- **구현은 새로 짜지 말고 기존 로컬 프롬프트바 컴포넌트를 가져와 공용화.**

## 8. AI 채팅 전용 페이지
헤더 "AI 채팅 · Claude 기반". 좌측: +새 대화 · 추천 질문 · 최근 대화. 메인: "무엇을 도와드릴까요?" + 추천 칩 + 프롬프트바. 푸터: 면책 문구. 역할: 자유 대화 + 새 작업 시작.

## 9. Workflow Lane (★ 보드의 핵심 — "하나 요청 → 연관 자료가 가로로 자라남")
My Board에서 교사가 한 번 요청하면, 채팅이 아니라 **가로로 펼쳐지는 워크플로 레인**이 생기고 단계 노드가 왼→오로 채워진다. 킨더버스의 중심 상호작용.

### 9.1 개념
- **Lane(레인)** = 타입이 있는 워크플로 컨테이너(자유 카드와 달리 내부가 단계 순서로 자동 정렬).
- **Step Node(단계 노드)** = 레인 안의 카드(아이디어 리스트 / 활동 사진 / 계획안 / 활동지 …). 각 노드는 §4 레지스트리 컴포넌트로 렌더.
- **진행은 클릭으로만**(자동 전체 실행 금지). 단계 완료 시 다음액션 칩 등장(기본 + 라우터 제안 §3 규칙7).
- **선택이 다음 단계 입력**: 노드에서 항목 복수선택 → 선택분만 다음 단계로(비용·범위 통제).
- **편집**: 모든 노드는 인라인 편집 + 프롬프트바 핸들링(선택=컨텍스트, §5.5).

### 9.2 놀이계획 템플릿 (기본 경로 + 동적 제안)
```
요청("5월 가정의 달 활동 추천")
└─ Lane: play_plan 생성
   ① 아이디어 리스트      (agent.plan)     → 복수선택
   ② 활동 사진(개념)      (agent.studio)   ※ AI 생성 일러스트 — 실제 아동사진 아님(라벨 표시)
   ③ 계획안               (agent.plan)     → 인라인/프롬프트바 편집
   ④ 활동지               (agent.studio)   → 편집·저장 (link.plan_id 연결)
   [+] 라우터 제안(suggested_next): 가정연계 통신문(agent.writing) 등 — 옅게 "추천"
   └─ [레인 저장] → 폴더에 번들(계획안+활동지+생성이미지+연결자료) 매니페스트로
```
- 기본 단계는 템플릿 정적 정의(순서·담당 에이전트·기본 다음액션). 라우터는 매 단계 맥락(완료 노드·선택·연령·계절)으로 `suggested_next`를 동적 생성.
- 데이터 축적 시 제안 정확도 상승(§8 자가고도화) — **제안 레이어는 초기부터 탑재**.

### 9.3 Workflow Runner (오케스트레이션 한 겹 — 새 에이전트 아님)
- 라우터가 "이건 play_plan 워크플로"로 인식 → 러너가 레인 생성.
- 러너는 템플릿 순서대로 **기존 전문 에이전트(plan/studio/writing/record)**를 호출해 결과를 노드 카드로 떨군다. 에이전트는 그대로, 러너가 흐름만 관장.
- 각 노드 산출은 §3 출력 계약 + `lane` 컨텍스트를 포함.

### 9.4 데이터 모델(개념)
```
workflow_lanes(id, board_id, template, title, status, created_at)
lane_nodes(id, lane_id, step, order, type, content_json, source_agent, links[])
lane_bundles(lane_id, folder_id, manifest_json)   // 저장 시 폴더 번들
```
- 레인 저장 = `lane_bundles`로 묶어 `폴더`에 1건으로. 폴더에서 레인 제목 하나로 전체 재오픈(계획안·활동지·이미지·연결자료 동반).

### 9.5 경계/안전
- ②의 활동 사진은 **AI 생성 개념 이미지** — 우리반/갤러리의 실제 아동 사진(동의·분류 트랙)과 분리하고 "AI 생성" 라벨 필수.
- 단계별 생성은 비용 발생 → 클릭 트리거 + 캐스케이드(§PRD 7). 발송·삭제는 자율성 게이트(L2/L3).
- 자유 보드 요소(스티키·텍스트)와 레인은 공존하되, 레인은 구조화 컨테이너로 구분 렌더.
